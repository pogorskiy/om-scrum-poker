### [72 → 50] Отсутствие backpressure на уровне Broadcast
- **Источник:** 04-architecture.md (находка #2)
- **Оригинальная severity:** 72 (HIGH)
- **Новая severity:** 50 (MEDIUM)
- **Аргумент:** Ревьюер сам пишет: "один медленный клиент не задержит остальных благодаря буферу в 32 сообщения". `Client.Send()` — неблокирующий (`select { case c.send <- msg: default: c.Close() }`). Broadcast копирует список клиентов и отпускает Lock перед отправкой. "Каскадный сценарий" теоретически возможен, но `Close()` только закрывает канал `done` через `closeOnce.Do(func(){ close(c.done) })` — это наносекундная операция, не блокирует ничего. `WritePump` завершится когда `ctx.Done()` или `c.done` сработает. Нет 5-секундного ожидания в Close — write timeout на уровне WritePump, не на уровне Broadcast.
- **Верификация:** `internal/server/client.go:59-65` — Send неблокирующий, Close через closeOnce мгновенный. `room_manager.go:112-126` — Broadcast корректно копирует список, отпускает Lock. HIGH severity не оправдан.

## Пропущенные проблемы

### [ПРОПУЩЕНО, 60] Отсутствие leave-события при закрытии вкладки на iOS Safari
- **Описание:** `disconnect()` вызывается в cleanup `useEffect` при unmount RoomPage. Однако на iOS Safari `beforeunload` и `unload` события ненадёжны. WebSocket `close` frame может не быть отправлен. Это значит, что участник останется в комнате как "active" до следующего ping timeout (10 секунд). Ни один ревьюер не упомянул платформо-специфичное поведение WebSocket при закрытии вкладки.
- **Рекомендация:** Сервер уже обрабатывает это через ping timeout в WritePump (5 секунд ping interval, ws.go client.go:113-124). При отсутствии pong — write error -> WritePump завершается -> readPump context cancelled -> disconnect flow. Проблема самоисцеляющаяся, но с задержкой.

### [ПРОПУЩЕНО, 55] Room.Join при rejoin сбрасывает роль на переданную клиентом
- **Описание:** При reconnect клиент отправляет `join` с `role: userRole.value`. Но если другой участник изменил роль этого пользователя (что невозможно в текущем UI, но возможно через API), rejoin перезапишет её. Более важно: `userRole` на клиенте — это локальный сигнал, а не серверное значение. Если пользователь открывает ту же комнату в новой вкладке, `userRole` будет дефолтным 'voter', даже если на сервере он 'observer'.
- **Верификация:** `web/src/state.ts:127` — `userRole = signal<'voter' | 'observer'>('voter')` — всегда 'voter' при загрузке. `internal/domain/room.go:219-229` — rejoin обновляет role на переданное значение.

### [ПРОПУЩЕНО, 45] Makefile build target не проверяет ошибки npm
- **Описание:** Ни один ревьюер не проверил Makefile. Архитект (#16) отметил отсутствие frontend тестов в make test, но не проверил сам build target.
