# Финальный отчёт — Комплексное ревью проекта om-scrum-poker
- Дата: 2026-04-08
- Статус: done
- Участники: Фронтенд, Бекенд, Безопасность, Архитектура, Продукт, Адвокат дьявола
- Модель: Claude Opus 4.6

## Общая оценка проекта

Проект om-scrum-poker представляет собой качественно реализованный легковесный инструмент для scrum poker с грамотным следованием KISS-принципу. Кодовая база чистая и хорошо структурированная: Go-бэкенд корректно использует конкурентные примитивы, фронтенд на Preact демонстрирует хорошую компонентную архитектуру с BEM-именованием, дизайн-токенами и базовой accessibility. Трёхуровневая система rate limiting, reconnect с exponential backoff и защита от XSS через JSX escaping — сильные стороны проекта.

Основные области риска сосредоточены в трёх направлениях. Во-первых, расхождение спецификации и реализации: 6 WebSocket-событий (timer, role) и несколько полей в payload не документированы, при этом спецификация заявлена как исчерпывающая. Во-вторых, конкурентность: несколько race conditions (disconnect flow, TOCTOU в handleJoin, dual-tab с одним sessionId) не критичны по отдельности, но создают класс трудноотлаживаемых багов. В-третьих, UX на мобильных устройствах и навигация: хедер ломается на маленьких экранах, отсутствует навигация назад из комнаты, деструктивные действия недостаточно защищены.

После корректировок адвоката дьявола критических проблем (90-100) не выявлено. Большинство HIGH-находок security-ревьюера были обоснованно понижены до MEDIUM с учётом контекста KISS-проекта без аутентификации для доверенных команд. Проект готов к использованию по назначению, но требует внимания к документации протокола и нескольким конкретным багам (spread при консенсусе, сортировка участников, очередь сообщений без лимита).

## Все находки (объединённые и отсортированные)

### [75] Двухуровневая блокировка: GC берёт эксклюзивный Lock на весь RoomManager
- **Severity:** HIGH
- **Источники:** Архитектура (находка #1)
- **Файл:** `internal/server/room_manager.go:53-66`
- **Проблема:** `collectGarbage()` берёт эксклюзивный `rm.mu.Lock()`, что блокирует ВСЕ операции с комнатами (GetRoom, Broadcast, RegisterClient) на время итерации по всем комнатам. При злонамеренном создании тысяч комнат GC может вызвать ощутимую задержку для всех пользователей.
- **Влияние:** Деградация производительности при большом количестве комнат. Deadlock невозможен (lock ordering соблюдается), но starvation реальна.
- **Рекомендация:** Разделить GC на две фазы: сбор кандидатов под RLock, удаление по одному под Lock с повторной проверкой.
- **Effort:** medium

### [75] Протокол WebSocket не документирован для timer и role событий
- **Severity:** HIGH
- **Источники:** Фронтенд (75), Бекенд (65)
- **Файл:** `docs/planning/04-architecture.md:387-411`, `internal/server/events.go`
- **Проблема:** 6 типов WebSocket-сообщений (`update_role`, `role_updated`, `timer_set_duration`, `timer_start`, `timer_reset`, `timer_updated`) полностью отсутствуют в спецификации, которая заявлена как исчерпывающая: "This table lists ALL WebSocket events." Также не документированы поля `createdBy`, `timer` в `room_state`, и `role` в `join` payload.
- **Влияние:** Новый разработчик или агент создаст несовместимую реализацию. При рефакторинге события могут быть случайно удалены.
- **Рекомендация:** Обновить `docs/planning/04-architecture.md` Section 4.3, добавив все 6 событий с payload-примерами. Обновить CLAUDE.md.
- **Effort:** low

### [72] Очередь сообщений (messageQueue) растёт без ограничений при отключении WebSocket
- **Severity:** HIGH
- **Источники:** Фронтенд (72), Безопасность (42, SEC-10), Архитектура (42, находка #12)
- **Файл:** `web/src/ws.ts:44-51`
- **Проблема:** При потере соединения `send()` складывает сообщения в `messageQueue` без лимита. При reconnect `flushQueue()` отправляет все залпом, что может превысить серверный rate limit (20 msg/sec) и привести к повторному отключению. Также отправляются устаревшие данные (старые голоса из прошлых раундов).
- **Влияние:** Self-DoS при reconnect после длительного отключения. Утечка памяти при активном UI без соединения.
- **Рекомендация:** Очищать очередь при reconnect, отправлять только `join` — `room_state` от сервера восстановит состояние.
- **Effort:** low

### [68] Race condition при disconnect: broadcast "disconnected" после UnregisterClient
- **Severity:** MEDIUM
- **Источники:** Архитектура (находка #3)
- **Файл:** `internal/server/ws.go:99-121`
- **Проблема:** Клиент удаляется из списка (`UnregisterClient`) ДО установки `Status = "disconnected"`. В окно между этими операциями другой клиент при rejoin получит `room_state` с участником в статусе "active", хотя тот уже отключён.
- **Влияние:** Кратковременная рассинхронизация состояния. Вероятность проявления мала (миллисекунды), но это реальный баг.
- **Рекомендация:** Переместить установку статуса и broadcast ДО UnregisterClient, или объединить в атомарную операцию.
- **Effort:** low

### [65] Room.Participants как публичное поле — нарушение инкапсуляции домена
- **Severity:** MEDIUM
- **Источники:** Архитектура (находка #4)
- **Файл:** `internal/domain/room.go`, `internal/server/ws.go:108-411`
- **Проблема:** Серверный пакет напрямую мутирует `Room.Participants` (прямая установка статуса, LastPing, итерация) в обход доменных методов. Это обходит валидацию и создаёт хрупкую связь между пакетами.
- **Влияние:** При рефакторинге домена потребуется менять серверный пакет. Возможна установка невалидного состояния.
- **Рекомендация:** Добавить методы `SetParticipantStatus()`, `UpdatePingTime()`, `GetParticipantSnapshot()`. Для KISS-проекта с двумя пакетами текущее состояние терпимо.
- **Effort:** medium

### [62] Timer: нет серверного уведомления при истечении — клиенты управляют переходом expired
- **Severity:** MEDIUM
- **Источники:** Архитектура (находка #5), Фронтенд (58)
- **Файл:** `web/src/components/Timer/Timer.tsx:29`, `internal/domain/room.go:436-444`
- **Проблема:** Сервер не запускает горутину для обратного отсчёта. Переход running -> expired происходит на клиенте. Разные клиенты могут показать expired в разные моменты (расхождение часов). Компонент Timer напрямую мутирует глобальный сигнал `timerState`, нарушая однонаправленный поток данных.
- **Влияние:** Рассинхронизация клиентов по состоянию таймера. Невозможность реализовать авто-reveal по таймеру без серверной горутины.
- **Рекомендация:** Для текущих требований (декоративный таймер) приемлемо. Использовать локальный state для displayRemaining вместо мутации глобального сигнала.
- **Effort:** medium

### [60] Отсутствие leave-события при закрытии вкладки на iOS Safari

### [55] TOCTOU race condition: проверка существования комнаты и создание в handleJoin
- **Severity:** MEDIUM
- **Источники:** Бекенд (72)
- **Файл:** `internal/server/ws.go:210-222`
- **Проблема:** Между `GetRoom()` (RLock) и `GetOrCreateRoom()` (Lock) мьютекс отпускается. Другая горутина может создать комнату в промежутке, вызвав ложный расход токена rate limiter.
- **Влияние:** Некорректный расход токенов rate limiter. Не security vulnerability.
- **Рекомендация:** Перенести проверку rate limit внутрь `GetOrCreateRoom` или создать атомарный метод.
- **Effort:** medium
- **Корректировка адвоката:** 72 -> 55. Последствия минимальны: один лишний токен rate limiter.

### [55] Room.Join при rejoin сбрасывает роль на переданную клиентом
- **Severity:** MEDIUM
- **Источники:** Адвокат дьявола (пропущенная проблема)
- **Файл:** `web/src/state.ts:127`, `internal/domain/room.go:219-229`
- **Проблема:** `userRole` на клиенте всегда 'voter' при загрузке. При reconnect или открытии в новой вкладке роль observer будет перезаписана на voter. Серверное состояние роли теряется.
- **Влияние:** Observer при reconnect становится voter. При открытии той же комнаты в новой вкладке роль сбрасывается.
- **Рекомендация:** Восстанавливать роль из `room_state` при получении, а не отправлять локальное значение.
- **Effort:** low

### [55] Отсутствие CORS-заголовков и HTTP security headers
- **Severity:** MEDIUM
- **Источники:** Архитектура (55, находка #7), Безопасность (72, SEC-03)
- **Файл:** `internal/server/handler.go:36-57`
- **Проблема:** Сервер не устанавливает X-Content-Type-Options, X-Frame-Options, Referrer-Policy. ALLOWED_ORIGINS влияет только на WebSocket, но не на HTTP-эндпоинты.
- **Влияние:** Для self-hosted KISS-проекта без PII риск минимален, но X-Content-Type-Options и X-Frame-Options — 5 строк кода.
- **Рекомендация:** Добавить middleware с минимальным набором: `X-Content-Type-Options: nosniff`, `X-Frame-Options: DENY`, `Referrer-Policy: same-origin`.
- **Effort:** low
- **Корректировка адвоката:** SEC-03: 72 -> 40. Объединённая оценка учитывает оба источника: 55.

## Сильные стороны проекта

- **Корректная конкурентность**: мьютексы используются последовательно, нет вложенных блокировок, sync.Once для Close, atomic для lastActivity
- **Трёхуровневая система rate limiting**: connection rate, concurrent connections, message rate — все корректно реализованы с cleanup goroutines
- **XSS-защита**: Preact JSX escaping, нигде нет dangerouslySetInnerHTML, серверная санитизация залго и control characters
- **Reconnect с exponential backoff и jitter**: 30% jitter, 10s cap, slow polling после 30s timeout, полное покрытие тестами
- **Modal на нативном dialog**: корректный фокус, backdrop-click, dismiss-режимы, aria-атрибуты
- **Система дизайн-токенов с тестами**: автоматическая проверка покрытия dark theme, отсутствия hardcoded hex
- **Embed-стратегия**: web/embed.go в отдельном пакете, fallback embedded -> disk -> placeholder
- **Чистая компонентная архитектура**: PascalCase dirs, BEM CSS, разделение presentation/container
- **prefers-reduced-motion**: flip-анимация отключается для accessibility
- **KISS-принцип**: минимум зависимостей (1 Go-зависимость), минимум абстракций, понятная кодовая база
