## Находки

### SEC-01. Отсутствие аутентификации и авторизации действий в комнате

**Оценка: 85 / 100 — HIGH**

**Файлы:** `internal/server/ws.go` (строки 159-188, dispatch), `internal/domain/room.go`

**Описание:**
Любой участник комнаты может выполнять привилегированные действия: `reveal`, `new_round`, `clear_room`. Нет понятия владельца комнаты или ролей с разделением прав. Поле `CreatedBy` в `Room` хранится, но нигде не проверяется для авторизации.

Злоумышленник, зная roomId (который виден в URL), может подключиться и:
- Очистить комнату (`clear_room`), удалив всех участников
- Раскрыть голоса преждевременно (`reveal`)
- Запустить новый раунд (`new_round`), сбросив все голоса

**Последствия:** Срыв рабочего процесса команды. Griefing-атака через очистку комнат.

**Рекомендация:**
Для легковесного сервиса без БД — добавить хотя бы проверку, что `clear_room` может вызвать только создатель комнаты (по sessionId). Или реализовать простой механизм «модератора» — первый участник получает расширенные права.

### SEC-02. Предсказуемость room ID позволяет перебор активных комнат

**Оценка: 75 / 100 — HIGH**

**Файлы:** `web/src/utils/room-url.ts` (строки 13-25), `internal/server/ws.go` (строки 68-76)

**Описание:**
Room ID формируется из slugified имени + 12 hex символов (6 байт = 48 бит энтропии). Regex валидации на сервере: `^[a-z0-9-]{1,64}$`. Хотя 48 бит исключают массовый перебор, имя комнаты в slug-части может быть предсказуемым (например, `sprint-review-<hex>`), сужая пространство поиска.

Более серьёзная проблема: **нет ограничения на количество попыток подключения к несуществующим комнатам**. Rate limiter проверяет только создание комнат (`AllowRoomCreation`), но при `join` в несуществующую комнату — комната автоматически создаётся через `GetOrCreateRoom`. Таким образом, атакующий может создавать мусорные комнаты, обходя rate limit для `AllowRoomCreation`, если комната уже существует.

**Последствия:** Возможен перебор комнат по известным паттернам имён. Засорение памяти мусорными комнатами.

**Рекомендация:**
- Увеличить случайную часть до 16+ байт (128 бит)
- Добавить rate limiting на неудачные попытки join (комната не найдена)
- Рассмотреть rate limit на общее количество комнат в системе

### SEC-03. Отсутствие HTTP-заголовков безопасности

**Оценка: 72 / 100 — HIGH**

**Файлы:** `internal/server/handler.go` (строки 36-57, NewServer)

**Описание:**
Сервер не устанавливает никаких security-заголовков:
- **Нет Content-Security-Policy (CSP)** — нет защиты от XSS через inline-скрипты
- **Нет X-Frame-Options / frame-ancestors** — приложение может быть встроено в iframe для clickjacking
- **Нет X-Content-Type-Options: nosniff** — браузер может неправильно интерпретировать MIME-тип
- **Нет Strict-Transport-Security (HSTS)** — нет принуждения к HTTPS
- **Нет Referrer-Policy** — URL комнаты (содержащий roomId) может утечь через Referer
- **Нет Permissions-Policy** — доступ к камере/микрофону/геолокации не ограничен

Для SPA, обслуживаемого из того же origin, это особенно важно, так как один сервер отвечает и за статику, и за API.

**Рекомендация:**
Добавить middleware с минимальным набором заголовков:
```go
w.Header().Set("X-Content-Type-Options", "nosniff")
w.Header().Set("X-Frame-Options", "DENY")
w.Header().Set("Referrer-Policy", "same-origin")
w.Header().Set("Content-Security-Policy", "default-src 'self'; connect-src 'self' wss: ws:; style-src 'self' 'unsafe-inline'")
```

### SEC-04. Session hijacking через предсказуемый sessionId

**Оценка: 70 / 100 — HIGH**

**Файлы:** `web/src/state.ts` (строки 96-110), `internal/server/ws.go` (строки 200-201)

**Описание:**
SessionId генерируется на клиенте (32 hex символа = 16 байт = 128 бит) и хранится в localStorage. Сервер принимает любой sessionId, соответствующий regex `^[a-f0-9]{32}$`, без дополнительной привязки.

Проблемы:
1. **Перехват сессии**: если атакующий узнает sessionId другого пользователя (например, через XSS на другом сайте, shared URL с sessionId в логах, или shoulder surfing), он может присоединиться к комнате от имени этого пользователя
2. **Нет привязки session к connection**: один и тот же sessionId может использоваться с разных IP/устройств одновременно. При `join` с существующим sessionId — происходит rejoin, обновляя имя и роль без подтверждения
3. **localStorage доступен всем скриптам на том же origin** — если когда-либо появится XSS-уязвимость, sessionId будет скомпрометирован

128 бит энтропии делают brute-force невозможным, но уязвимость в архитектуре остаётся.

**Рекомендация:**
- Привязывать sessionId к серверному токену при первом join (выдавать одноразовый connection token)
- Рассмотреть генерацию sessionId на сервере

### SEC-05. Раскрытие информации через /health endpoint

**Оценка: 55 / 100 — MEDIUM**

**Файл:** `internal/server/handler.go` (строки 59-78)

**Описание:**
Endpoint `/health` доступен без аутентификации и возвращает:
- Количество активных комнат (`rooms`)
- Количество соединений (`connections`)
- Количество WebSocket-соединений (`active_websockets`)
- Время работы сервера (`uptime`)
- Время сборки (`build_time`)

Эта информация полезна атакующему для:
- Определения загруженности сервера (выбор момента для DoS)
- Понимания масштаба использования приложения
- Fingerprinting версии по build_time

**Рекомендация:**
- Ограничить `/health` до внутренней сети или требовать API-ключ для расширенной информации
- Возвращать только `{"status":"ok"}` для публичного доступа

### SEC-06. Отсутствие ограничения размера тела HTTP-запроса

**Оценка: 52 / 100 — MEDIUM**

**Файл:** `internal/server/handler.go` (строки 50-56)

**Описание:**
HTTP-сервер настроен с `ReadHeaderTimeout: 5s` и `IdleTimeout: 60s`, но нет `ReadTimeout` или `WriteTimeout` на уровне HTTP-сервера. Для WebSocket это нормально (долгоживущие соединения), но для обычных HTTP-запросов это создаёт вектор для slowloris-атаки.

Хотя WebSocket-сообщения ограничены `maxMessageSize = 4096`, HTTP-запросы к `/health` и `/` не имеют ограничений на тело запроса (хотя они и не читают тело).

**Рекомендация:**
- Добавить `ReadTimeout` для не-WebSocket эндпоинтов
- Рассмотреть использование отдельных серверов для HTTP и WS

### SEC-07. Race condition при одновременном join с одним sessionId

**Оценка: 50 / 100 — MEDIUM**

**Файлы:** `internal/server/ws.go` (строки 190-269), `internal/server/room_manager.go` (строки 94-101)

**Описание:**
Два клиента могут одновременно отправить `join` с одинаковым sessionId в одну комнату. Хотя `room.Join()` вызывается под `room.Lock()`, `manager.RegisterClient()` регистрирует оба клиента в `clients` map. В результате оба клиента получают сообщения, но сервер считает их одним участником.

При отключении одного клиента — `UnregisterClient` удалит его из clients map, а в `HandleWebSocket` будет установлен статус `disconnected` для общего sessionId, хотя второй клиент всё ещё активен.

**Последствия:** Некорректное состояние присутствия, фантомные сообщения, возможность подслушивания чужой сессии.

**Рекомендация:**
- При повторном join с тем же sessionId — отключать предыдущее соединение
- Добавить connection ID, независимый от session ID

### SEC-08. Потенциальная утечка памяти через создание мусорных комнат

**Оценка: 48 / 100 — MEDIUM**

**Файлы:** `internal/server/room_manager.go` (строки 53-66, collectGarbage), `internal/server/ws.go` (строки 210-216)

**Описание:**
Garbage collector удаляет комнаты только через 24 часа (`roomExpiry`) после последней активности И при отсутствии подключённых клиентов. Атакующий может:
1. Создавать комнаты с уникальными ID (rate limit: 5/мин на IP)
2. Через разные IP или за длительное время — создать тысячи комнат
3. Каждая комната потребляет память (map, Timer, metadata)

За 24 часа с одного IP можно создать 5 * 60 * 24 = 7200 комнат. С ботнета — значительно больше.

Нет верхнего ограничения на общее количество комнат в системе.

**Рекомендация:**
- Добавить глобальный лимит на количество комнат (например, `MAX_ROOMS`)
- Уменьшить `roomExpiry` для пустых комнат (например, 1 час вместо 24)
- Добавить мониторинг роста количества комнат

### SEC-09. Отсутствие валидации roomName и createdBy на сервере

**Оценка: 45 / 100 — MEDIUM**

**Файлы:** `internal/domain/room.go` (строки 120-138, NewRoom), `internal/server/ws.go` (строки 218-223)

**Описание:**
При создании комнаты `roomName` обрезается до 60 символов (`MaxRoomName`), но не проходит санитизацию `sanitizeName()`. Поле `createdBy` вообще не валидируется — оно устанавливается из `p.UserName` в `handleJoin`, которое прошло `sanitizeName`, но `roomName` из payload напрямую:

```go
roomName := p.RoomName
if roomName == "" {
    roomName = p.UserName + "'s Room"
}
room, err := manager.GetOrCreateRoom(client.roomID, roomName, p.UserName)
```

`roomName` не проходит через `sanitizeName()` и может содержать:
- Unicode control characters
- Zalgo text
- Zero-width characters

Хотя Preact по умолчанию экранирует HTML при рендеринге через JSX `{state.roomName}`, непроверенные данные могут быть проблемой при отладке/логировании.

**Рекомендация:**
- Применить `sanitizeName()` к `roomName` перед сохранением
- Рассмотреть отдельную функцию санитизации для имён комнат

### SEC-10. WebSocket message flooding через message queue

**Оценка: 42 / 100 — MEDIUM**

**Файлы:** `internal/server/client.go` (строки 46-66), `web/src/ws.ts` (строки 44-51)

**Описание:**
На фронтенде `messageQueue` не имеет ограничения размера. При потере соединения все отправленные сообщения копятся в массиве и отправляются при переподключении (`flushQueue`). Злонамеренный или сбойный клиент может накопить большое количество сообщений.

На бэкенде `sendBufferSize = 32` — при переполнении буфера клиент отключается, что корректно. Однако при `flushQueue` все накопленные сообщения отправляются мгновенно, потенциально превышая rate limit (`DefaultMsgRateLimiter: 20 msg/sec`), что приведёт к отключению.

**Последствия:** Потеря сообщений при reconnect, self-DoS.

**Рекомендация:**
- Ограничить размер `messageQueue` на фронтенде (например, 50 сообщений)
- Добавить throttling при `flushQueue`

### SEC-11. X-Forwarded-For spoofing при TRUST_PROXY=true

**Оценка: 40 / 100 — MEDIUM**

**Файл:** `internal/server/ws.go` (строки 619-635, clientIP)

**Описание:**
При `TRUST_PROXY=true` сервер доверяет первому значению из `X-Forwarded-For`. Это стандартный подход, но:
1. Если между клиентом и сервером нет доверенного прокси — атакующий может подделать IP
2. Используется `parts[0]` (первый IP в цепочке) — это IP, указанный клиентом, а не добавленный прокси

Правильный подход — брать последний IP перед доверенным прокси (rightmost-first), а не первый.

**Последствия:** Полный обход rate limiting и connection tracking по IP.

**Рекомендация:**
- Документировать, что `TRUST_PROXY` безопасен только за обратным прокси
- Реализовать rightmost-first подход для `X-Forwarded-For`
- Рассмотреть поддержку списка доверенных прокси

### SEC-12. Отсутствие валидации origin при пустом ALLOWED_ORIGINS

**Оценка: 38 / 100 — LOW**

**Файл:** `internal/server/ws.go` (строки 39-49, buildAcceptOptions)

**Описание:**
При пустом `ALLOWED_ORIGINS` используются дефолтные опции библиотеки `nhooyr.io/websocket`, которая проверяет same-origin. Это корректное поведение. Однако при значении `*` используется `InsecureSkipVerify: true`, что полностью отключает проверку origin.

Документация не подчёркивает, что `*` — это небезопасная настройка для продакшена. Если администратор установит `ALLOWED_ORIGINS=*` для «удобства» — все cross-origin атаки станут возможны.

**Рекомендация:**
- Логировать предупреждение при `ALLOWED_ORIGINS=*`
- В документации явно указать риски
- Рассмотреть дополнительный CSRF-токен для WebSocket handshake

### SEC-13. Отсутствие логирования событий безопасности

**Оценка: 35 / 100 — LOW**

**Файлы:** `internal/server/ws.go`, `internal/server/ratelimit.go`

**Описание:**
Логируются только технические ошибки (read/write errors, ping errors). Не логируются:
- Срабатывания rate limiter (кроме message rate limit на строке 145)
- Отклонённые подключения по лимиту соединений
- Неудачные попытки join (невалидный roomId, sessionId)
- Подозрительная активность (массовое создание комнат)

Без логов безопасности невозможно:
- Обнаружить атаку в реальном времени
- Провести post-mortem инцидента
- Настроить алертинг

**Рекомендация:**
- Добавить structured logging для всех security-значимых событий
- Включить IP адрес, roomId, sessionId в логи при rate limiting
- Рассмотреть уровни логирования (INFO для нормальных событий, WARN для rate limiting)

### SEC-14. Клиентская валидация имени не соответствует серверной

**Оценка: 30 / 100 — LOW**

**Файлы:** `web/src/components/NameEntryModal/NameEntryModal.tsx` (строка 22), `web/src/components/EditNameModal/EditNameModal.tsx` (строка 28), `internal/domain/room.go` (строки 153-198, 200-210)

**Описание:**
Фронтенд ограничивает имя только `maxLength={30}` и проверкой `trim() !== ''`. Сервер выполняет сложную санитизацию: удаление control characters, zero-width символов, ограничение combining marks. После санитизации имя может измениться — клиент не знает о результате, пока не получит `name_updated`.

Нет клиентской валидации на:
- HTML-символы (хотя Preact экранирует при рендере)
- Unicode control characters
- Чрезмерные combining marks (zalgo text)

**Последствия:** UX-проблема — пользователь видит одно имя на вводе, другое после отправки.

**Рекомендация:**
- Перенести базовую санитизацию на фронтенд
- Показывать пользователю результат санитизации до отправки

### SEC-15. localStorage не очищается при выходе

**Оценка: 25 / 100 — LOW**

**Файл:** `web/src/state.ts` (строки 80-94, 104-110)

**Описание:**
SessionId и userName хранятся в localStorage бессрочно. На shared/public компьютерах это означает:
- Следующий пользователь получает доступ к sessionId предыдущего
- Может подключиться к тем же комнатам от имени предыдущего пользователя

Нет функции «выход» или очистки данных сессии.

**Рекомендация:**
- Добавить кнопку «Выход» с очисткой localStorage
- Рассмотреть использование sessionStorage вместо localStorage
- Добавить таймаут сессии (24ч)

### SEC-16. Нет ограничения на количество участников-наблюдателей

**Оценка: 22 / 100 — LOW**

**Файл:** `internal/domain/room.go` (строки 232-234)

**Описание:**
`MaxParticipants = 50` ограничивает общее число участников. Но все 50 могут быть наблюдателями, что позволяет заполнить комнату и заблокировать вход для голосующих. Нет отдельного лимита для наблюдателей.

**Рекомендация:**
- Рассмотреть раздельные лимиты для voter/observer
- Или не считать observers в лимит

### SEC-17. Docker-образ на базе scratch без TLS

**Оценка: 20 / 100 — LOW**

**Файл:** `Dockerfile` (строки 19-23)

**Описание:**
Финальный образ на `scratch` — это хорошая практика (минимальная поверхность атаки). Однако:
- Нет поддержки TLS в приложении (нет опций для сертификатов)
- В scratch нет корневых CA-сертификатов (если приложение когда-либо будет делать исходящие HTTPS-запросы)
- Нет пользователя (работает от root, хотя в scratch это минимальный риск)

**Рекомендация:**
- Документировать необходимость TLS-терминации на reverse proxy
- Добавить опцию для TLS (CERT_FILE, KEY_FILE)

### SEC-18. Preact обеспечивает защиту от XSS по умолчанию

**Оценка: 10 / 100 — NOTE (позитивная находка)**

**Файлы:** все `*.tsx` компоненты

**Описание:**
Все пользовательские данные (userName, roomName, vote values, toast messages) рендерятся через JSX-выражения `{variable}`, которые Preact автоматически экранирует. Нигде не используется `dangerouslySetInnerHTML`. Это обеспечивает надёжную защиту от stored/reflected XSS.

Единственное место с потенциальным риском — динамические CSS-классы (например, `participant-card__status--${participant.status}`), но `status` валидируется на сервере как `"active" | "idle" | "disconnected"`.

**Статус:** Защита XSS реализована корректно.

### SEC-19. WebSocket message size limit реализован корректно

**Оценка: 8 / 100 — NOTE (позитивная находка)**

**Файл:** `internal/server/ws.go` (строка 83)

**Описание:**
`conn.SetReadLimit(maxMessageSize)` = 4096 байт. Библиотека `nhooyr.io/websocket` автоматически закрывает соединение при превышении лимита. Это надёжная защита от memory exhaustion через большие WS-сообщения.

### SEC-20. Корректная реализация rate limiting на уровне соединений

**Оценка: 5 / 100 — NOTE (позитивная находка)**

**Файлы:** `internal/server/ratelimit.go`, `internal/server/conntracker.go`, `internal/server/msg_rate_limiter.go`

**Описание:**
Трёхуровневая система rate limiting:
1. **Connection rate** — 20 WS-подключений/мин на IP (token bucket)
2. **Concurrent connections** — 100/IP, 1000 глобально
3. **Message rate** — 20 msg/sec на соединение (per-connection token bucket)

Все три уровня корректно реализованы с proper mutex locking. Cleanup goroutine предотвращает утечку памяти в rate limiter.

### Общая оценка безопасности

**Для заявленного scope** (легковесный self-hosted инструмент без аутентификации для внутренних команд) приложение имеет **приемлемый уровень безопасности** с несколькими значимыми пробелами.

**Сильные стороны:**
- Корректная защита от XSS через Preact JSX escaping
- Трёхуровневый rate limiting
- Ограничение размера WS-сообщений
- Строгая валидация roomId и sessionId по regex
- Санитизация имён пользователей (zalgo, control chars)
- Input validation для vote values (whitelist)
- Минимальный Docker-образ (scratch)

**Критические пробелы:**
- Отсутствие авторизации действий (любой участник может очистить комнату)
- Отсутствие HTTP security headers
- X-Forwarded-For spoofing при неправильной конфигурации
- Отсутствие логирования событий безопасности

**Приоритетные действия:**
1. Добавить HTTP security headers (SEC-03) — быстрая победа, минимальные усилия
2. Добавить авторизацию для деструктивных действий (SEC-01) — средние усилия, высокий импакт
3. Исправить X-Forwarded-For parsing (SEC-11) — быстрая победа
4. Добавить логирование security-событий (SEC-13) — средние усилия, необходимо для операционной безопасности
