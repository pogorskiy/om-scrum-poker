### [72] TOCTOU race condition: проверка существования комнаты и её создание в handleJoin

**Файл:** `internal/server/ws.go`, строки 210-222

**Описание:** В `handleJoin` сначала проверяется существование комнаты через `manager.GetRoom()` (берёт RLock), затем, если комнаты нет, проверяется rate limit, и только потом вызывается `GetOrCreateRoom()` (берёт Lock). Между `GetRoom()` и `GetOrCreateRoom()` мьютекс отпускается, и другая горутина может создать комнату в этом промежутке. В результате rate limit на создание комнаты может быть списан, хотя комната уже была создана другим клиентом.

```go
existing := manager.GetRoom(client.roomID) // RLock/RUnlock
if existing == nil {
    if !limiter.AllowRoomCreation(ip) {  // токен списан
        ...
    }
}
room, err := manager.GetOrCreateRoom(...)  // Lock/Unlock -- комната может уже существовать
```

**Последствия:** Некорректный расход токенов rate limiter'а. Не критично для безопасности, но может приводить к ложным срабатываниям ограничения.

**Рекомендация:** Перенести проверку rate limit внутрь `GetOrCreateRoom` или добавить метод, который атомарно проверяет существование + лимит + создание.

### [70] Несоответствие протоколу: maxMessageSize 4 KB вместо 1 KB по спецификации

**Файл:** `internal/server/ws.go`, строка 22

**Описание:** Спецификация (04-architecture.md, Section 4.1) однозначно указывает: "Max message size: 1 KB (inbound). Messages exceeding this close the connection." В коде установлено `maxMessageSize = 4096` (4 KB).

**Последствия:** Сервер принимает сообщения в 4 раза больше специфицированного. Это увеличивает поверхность атаки для payload-abuse, хотя реальный ущерб минимален из-за простоты протокола.

**Рекомендация:** Либо уменьшить до 1024 (привести в соответствие со спецификацией), либо обновить спецификацию до 4 KB с обоснованием (например, длинные имена в UTF-8).

### [62] handleClearRoom: broadcast отправляется до фактической очистки

**Файл:** `internal/server/ws.go`, строки 379-387

**Описание:** `room_cleared` broadcast отправляется до того, как `room.ClearRoom()` выполнен. Комментарий гласит: "Broadcast before clearing so all connected clients receive the event." Однако это создаёт окно, в котором получивший `room_cleared` клиент может отправить запрос (например, `vote`) и обработать его по старому состоянию комнаты, до её очистки.

```go
manager.Broadcast(client.roomID, msg)  // broadcast room_cleared
room.Lock()
room.ClearRoom()                       // фактическая очистка
room.Unlock()
```

**Последствия:** Возможна кратковременная рассинхронизация. На практике маловероятна из-за природы WebSocket и однопоточной обработки сообщений на клиенте.

**Рекомендация:** Переместить broadcast после `ClearRoom()`, либо задокументировать осознанный выбор.

### [58] Потенциальная утечка горутины при остановке WritePump

**Файл:** `internal/server/client.go`, строки 92-127; `internal/server/ws.go`, строки 91-98

**Описание:** В `HandleWebSocket` запускается горутина с `WritePump`, которая вызывает `cancel()` при завершении. Основной readPump тоже блокирует. Если readPump завершается раньше (ошибка чтения), выполняется `client.Close()` и `conn.Close(...)`. Однако `WritePump` может зависнуть в `conn.Write()` или `conn.Ping()` после закрытия соединения, пока не сработает контекст.

При этом `cancel()` вызывается в defer основной горутины, что отменяет контекст и позволяет WritePump завершиться. Это корректно. Но есть тонкость: `client.Close()` вызывается до `cancel()` (defer порядок LIFO, но cancel -- первый defer, Close вызывается в теле). WritePump мог бы попытаться писать в закрытый канал -- но `done` канал закрывается через `closeOnce`, и select в WritePump это ловит. В целом, механизм работает, но зависит от неочевидного порядка.

**Рекомендация:** Добавить комментарий, объясняющий жизненный цикл горутин и порядок очистки. Рассмотреть явный вызов `cancel()` перед `client.Close()`.

### [55] IPv6 парсинг в clientIP некорректен для bracket-нотации

**Файл:** `internal/server/ws.go`, строки 619-635

**Описание:** Функция `clientIP` использует `strings.LastIndex(addr, ":")` для отсечения порта. Для IPv6-адресов Go http.Server записывает RemoteAddr в формате `[::1]:8080`. `LastIndex(":")` корректно найдёт последнее двоеточие (порт), но результат будет `[::1]` -- с квадратными скобками. Тест подтверждает: `"[::1]:8080"` -> `"[::1]"`.

Это приводит к тому, что rate limiter и conn tracker используют ключ `[::1]` вместо `::1`. Хотя это консистентно (один и тот же клиент всегда получит один и тот же ключ), это не соответствует стандартному представлению IPv6 и может создать путаницу в логах и при сравнении с другими системами.

**Рекомендация:** Использовать `net.SplitHostPort` для корректного парсинга как IPv4, так и IPv6.

### [52] Spread всегда не-null при наличии числовых голосов, вопреки спецификации

**Файл:** `internal/domain/stats.go`, строки 87-89

**Описание:** Спецификация (Section 4.5, `votes_revealed`) гласит: "`spread` is `[min, max]` of numeric votes, or `null` if < 2 distinct numeric votes (including when all numeric votes are the same value, i.e. consensus)." Однако код всегда устанавливает spread при наличии числовых значений, даже при консенсусе (все одинаковые):

```go

result.Spread = &spread  // всегда non-null, даже при [5, 5]
```

**Последствия:** Фронтенд может отображать `spread: [5, 5]` вместо скрытия spread при консенсусе.

**Рекомендация:** Добавить проверку: если `hasConsensus` или `len(numericValues) < 2`, `spread` должен быть `nil`.

### [48] Канал send в Client не закрывается явно

**Файл:** `internal/server/client.go`

**Описание:** Канал `send` создаётся в `NewClient`, но нигде не закрывается через `close(c.send)`. `Client.Close()` закрывает только `done`. `WritePump` проверяет `case msg, ok := <-c.send: if !ok { return }`, но этот case никогда не сработает, потому что канал не закрывается.

Это не утечка в текущей реализации, потому что `WritePump` также слушает `c.done` и `ctx.Done()`. Однако семантически это неряшливо: паттерн с `ok` проверкой предполагает закрытие канала, которого не происходит.

**Рекомендация:** Либо закрывать `c.send` в `Close()`, либо удалить проверку `ok` из WritePump, чтобы код не вводил в заблуждение.

### [45] Отсутствие WriteTimeout на HTTP-сервере

**Файл:** `internal/server/handler.go`, строки 50-57

**Описание:** HTTP-сервер настроен с `IdleTimeout` (60s) и `ReadHeaderTimeout` (5s), но `WriteTimeout` не установлен (по умолчанию 0 = без ограничений). **Рекомендация:** Для WebSocket-сервера WriteTimeout обычно не устанавливают (это сломает долгоживущие соединения). Поведение корректно. Можно добавить комментарий, поясняющий осознанный выбор.

### [42] MsgRateLimiter не учитывает восстановление после rate limit

**Файл:** `internal/server/ws.go`, строки 143-147; `internal/server/msg_rate_limiter.go`

**Описание:** Когда клиент превышает лимит сообщений, readPump отправляет ошибку и немедленно завершается (`return`), что приводит к отключению клиента. Это жёсткая стратегия: один всплеск сообщений (больше 20 за раз) приводит к разрыву соединения. Альтернатива -- отправить предупреждение и продолжить, давая клиенту шанс замедлиться.

**Последствия:** В нормальных условиях 20 msg/sec более чем достаточно. Но при reconnect клиент может послать несколько сообщений быстро (join + vote + presence), и если одновременно тикер отправляет пинги, burst может быть неожиданно высоким. 20 сообщений -- достаточный запас.

**Рекомендация:** Текущая стратегия разумна для защиты от злоупотреблений. Можно добавить логирование IP-адреса при срабатывании для мониторинга.

### [40] Двойная десериализация JSON в MakeEnvelope

**Файл:** `internal/server/events.go`, строки 112-122

**Описание:** `MakeEnvelope` сначала сериализует payload в `[]byte`, затем оборачивает в Envelope и сериализует снова. Payload кодируется дважды: `json.Marshal(payload)` -> `json.RawMessage` -> `json.Marshal(envelope)`. Это стандартный паттерн для работы с `json.RawMessage`, но для broadcast-сценариев (одно сообщение -> N клиентов) это означает, что сериализация происходит один раз, что уже оптимально.

**Рекомендация:** Текущая реализация корректна и достаточно эффективна. Оставить как есть.

### [38] RateLimiter.cleanup и RateLimiter.Close: возможная паника при двойном Close

**Файл:** `internal/server/ratelimit.go`, строки 50-52

**Описание:** `Close()` вызывает `close(rl.stop)`. Если `Close()` вызван дважды, будет паника на повторном закрытии канала. В текущем коде `Close()` вызывается только в `defer` в `main.go`, что безопасно. Но API не защищён от двойного вызова.

**Рекомендация:** Добавить `sync.Once` для защиты, аналогично `Client.closeOnce`.

### [32] BuildRoomState сортирует участников по sessionId, а не по времени присоединения

**Файл:** `internal/server/room_manager.go`, строки 231-233

### [22] Нет graceful drain для новых подключений при shutdown

**Файл:** `cmd/server/main.go`, строки 56-66

**Описание:** При получении сигнала SIGINT/SIGTERM:
1. Вызывается `manager.CloseAll()` (закрывает WS-соединения)
2. Вызывается `srv.Shutdown(ctx)` (прекращает приём новых HTTP-запросов)

Порядок логичнее был бы обратный: сначала прекратить приём новых соединений (`srv.Shutdown`), а затем закрыть существующие WebSocket-соединения. При текущем порядке возможен сценарий: закрыли WS -> клиент переподключился (Shutdown ещё не вызван) -> Shutdown закрывает новое соединение некорректно.

**Рекомендация:** Изменить порядок: сначала `srv.Shutdown()`, затем `manager.CloseAll()`. Или вызывать `CloseAll()` из shutdown hook.

### [15] Хорошие практики (положительные наблюдения)

- **Конкурентность:** Мьютексы используются последовательно и корректно. Room имеет свой мьютекс, RoomManager -- свой. Нет вложенных блокировок (no lock ordering issues).
- **sync.Once:** Используется в `Client.Close()` для предотвращения двойного закрытия `done` канала.
- **atomic.Int64:** `lastActivity` в Room использует атомарные операции, что позволяет обновлять без полной блокировки.
- **Broadcast копирует список клиентов:** Мьютекс освобождается перед отправкой, что предотвращает deadlock.
- **Token bucket rate limiter:** Собственная реализация корректна, с периодической очисткой устаревших buckets.
- **Input validation:** Session ID, Room ID, имена участников -- все валидируются.
- **closeOnce + done channel:** Корректный паттерн завершения горутин.
- **Тесты:** Хорошее покрытие для handler, room_manager, client, events, ratelimit, conntracker. Тесты на конкурентность присутствуют.
